================================================================================
  一、S3 叙事层技术概述
================================================================================

  Stage 3 将 Stage 2 的 ST-Graph JSON 转换为自然语言交通报告。

  架构:
    ST-Graph JSON → 事实序列化 → LLM 推理 → 护栏检查 → 报告输出

  3.1 事实序列化 (serialize_graph_to_facts)
    从 ST-Graph 中聚合提取:
    - 车辆总数、类型分布 (Counter 统计)
    - 转向语义统计 (左转/右转/直行/静止, Counter)
    - 纵向趋势 (加速/减速/巡驶/蠕行, Counter)
    - 仅列出转弯车辆 (直行车辆不逐辆列出，减少冗余)

  3.2 LLM 推理
    模型: Ollama qwen2.5:7b (Q4_K_M 量化, 4.7GB)
    推理模式: 无 thinking 模式 (减少幻觉)
    超时: 300s
    重试: 最多 2 次

  3.3 护栏机制 (guardrail_check)
    12 个禁止关键词扫描，含否定词感知:
    - 禁止: "碰撞"、"事故"、"伤亡"、"危险"、"违规"等
    - 若事实中无异常事件但 LLM 编造 → 触发 re-prompt
    - re-prompt 注入更强禁令，强制只基于事实

  3.4 交互式问答 (可选)
    RAG 风格: 从 ST-Graph 检索相关事实 → 注入 prompt → LLM 回答
    无状态: 每次问答独立，不积累上下文

================================================================================
  五、S3 设计要点 (供报告/PPT文字使用)

================================================================================

  1. 为什么用 qwen2.5:7b 而非更大模型?
     - 本地部署需求 (Ollama, 无网环境可用)
     - RTX 4060 8GB 显存限制
     - 7B 模型对结构化事实→叙述的转换足够
     - 实测 qwen3.5:4b 输出过于简化, qwen2.5:7b 平衡最好

  2. 为什么关闭 thinking 模式?
     - thinking 模式 (qwen3.5) 会产生大量推理链
     - 推理链中容易出现"假设性"推断 → 触发护栏
     - 关闭后输出更直接、更忠实于输入事实

  3. 护栏的必要性
     - LLM 天然倾向"补充常识" — 会编造"可能发生的事故"
     - 本项目是可信系统，禁止幻觉
     - 12 个关键词覆盖交通事故、伤亡、违规等敏感词
     - 否定词感知: "没有发生碰撞" 不会被误判
