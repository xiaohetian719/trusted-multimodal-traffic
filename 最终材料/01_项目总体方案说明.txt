================================================================================
  一、项目背景与目标
================================================================================

本项目的目标是构建一套"感知与认知完全解耦"的交通AI系统。输入为交通监控
摄像头视频（AICity22 数据集），输出为可信的自然语言交通报告。

区别于端到端黑盒方案（如直接让多模态LLM看视频），本系统将视觉理解严格分层：
  - Stage 1（感知层）：只输出结构化数值数据，不做任何语义判断
  - Stage 2（拓扑层）：纯数学/几何推理，生成时空因果图 JSON
  - Stage 3（叙事层）：LLM 仅做"文书"工作，严格基于 JSON 事实撰写报告

这样设计的根本原因：防止 LLM 的视觉幻觉污染下游推理。LLM 看视频时会"脑补"
不存在的事件（碰撞、违章等），而我们要求报告 100% 可追溯到结构化事实。

================================================================================
  二、核心设计原则
================================================================================

1. 感知与认知解耦
   - 三阶段之间通过标准化 JSON/Pickle 交接，互不依赖
   - Stage 1 不对画面做任何语义解释，只输出 BBox + 深度值
   - Stage 2 使用纯数学方法（微分几何、FSM、LCSS），不调用任何模型
   - Stage 3 的 LLM 永远不直接访问视频帧

2. 绝对可信
   - 报告中的每一句话必须能在 ST-Graph JSON 中找到对应事实
   - 护栏（Guardrail）机制：LLM 输出经关键词扫描，若检测到"碰撞/事故"
     等编造词汇，自动触发重新生成
   - 遵循"面向结果编程"：不把所有数据丢给 LLM，而是先做聚合统计

3. 白盒开源 + 零微调
   - 全部使用开源预训练模型，不做任何深度学习训练
   - YOLO (BDD100K预训练), Depth Anything v2, Qwen2.5 (Ollama)

--------------------------------------------------------------------------------
  Stage 1 — 双通道感知层 (perception_pipeline.py, v4)
--------------------------------------------------------------------------------

  通道 B1: YOLO 目标检测 + 内置追踪
    - 模型: YOLO (BDD100K 预训练权重)
    - 仅保留车辆类别 (car/truck/bus/train, cls_id ∈ {2,3,4,5})
    - 内置追踪 persist=True，跨帧保持 track_id
    - 输出: 每个目标的 BBox 中心点 (cx, cy)、类别、置信度

  通道 C: 单目深度估计
    - 模型: Depth Anything v2 ViT-S (~95MB，轻量版)
    - 逐帧生成全图相对深度图
    - 在每个车辆 BBox 中心点 (cx, cy) 处采样深度值
    - 2D 像素空间 + 相对深度（不做 3D 投影，避免非线性畸变）

  输出 Frame Dict:
    {
      "frame_id": int,
      "timestamp": float,
      "objects": [
        {"id": int, "cls_id": int, "class": str, "conf": float,
         "cx": int, "cy": int, "depth": float}
      ]
    }

--------------------------------------------------------------------------------
  Stage 2 — 时空拓扑与因果推断层 (topology_engine.py, v2 四阶段)
--------------------------------------------------------------------------------

  Phase 1: 双数据池分流
    - 目的: 把可靠的完整轨迹和零碎轨迹分开
    - 规则: N≥15帧 且 平均置信度≥0.4 → stable_tracks（稳定池）
           否则 → suspended_fragments（待复检碎片池）
    - 关键参数: min_frames=15, conf_thresh=0.4

  Phase 2: LCSS 级联缝合
    - 目的: 把 Phase 1 中被切断的轨迹重新拼接
    - 方法: 最长公共子序列 (LCSS) 形态学比对 + 匈牙利算法全局指派
    - 关键参数: max_frame_gap=150, lcss_eps=30px, sim_thresh=0.8
    - 输出: 缝合后的完整轨迹 ID 链

  Phase 3: 轨迹数值精炼
    - 三步处理:
      ① 盲区线性插值 (填充检测丢失的帧，插值帧 conf×0.5)
      ② 15帧 uniform 权重滑动平滑 (替代旧的 30帧 conf 加权)
      ③ 长周期静止检测 (95% 阈值，替代旧的 70%)
    - 15帧窗口的选用依据: 平衡去噪效果和响应速度 (1.5s @10fps)
    - 95%阈值的选用依据: 只有真正从头到尾静止的车辆才标记为 Stationary

  Phase 4: 曲率积分 M2 转向语义
    - 核心公式: W = Σ ω_t
      其中 ω_t = (x'·y'' - y'·x'') / (x'² + y'²)   (曲率，逐帧微分几何)
    - 自适应阈值: ε = P85(|W|)  (取所有轨迹曲率绝对值的第85百分位)
      钳位区间: [0.12, 1.5] rad  ([6.9°, 85.9°])
    - 分类规则 (图像坐标系 y↓):
      |W| ≤ ε  → Go Straight (直行)
      W > +ε   → Right Turn  (右转，图像中向下弯曲)
      W < -ε   → Left Turn   (左转，图像中向上弯曲)
    - 总位移 < 20px → Stationary Override (静止覆盖)

  输出 ST-Graph JSON:
    {
      "meta_info": {总帧数, 时长, phase_summary},
      "nodes": {
        "car_XXXXX": {
          "type": "vehicle",
          "first_seen": "0.00s", "last_seen": "14.10s",
          "turn_semantic": "Left Turn" | "Right Turn" | "Go Straight",
          "longitudinal_trend": "Approaching" | "Departing" | "Stationary",
          "following": null | "car_YYYYY",
          "final_state": "Accelerating" | "Cruising" | "Decelerating" | "Creeping" | "Stationary"
        }
      },
      "temporal_event_chains": [...]
    }

--------------------------------------------------------------------------------
  Stage 3 — 可信文本叙事层 (narrative_agent.py, v2 精简)
--------------------------------------------------------------------------------

  模块 1: 事实序列化 (serialize_graph_to_facts)
    - 输入: Stage 2 的 ST-Graph JSON (可能有数百辆车)
    - 策略: 聚合统计 (Counter) + 仅列出转弯车辆
    - 不喂给 LLM 的内容: 全量 134 辆车逐一列出、longitudinal_trend（不可靠）
    - 结果: ~18 行精简事实，从 11k chars 压缩到 ~1k chars

  模块 2: LLM 推理 (Ollama Qwen2.5:7b)
    - 选型依据: Qwen3.5 有思考模式，推理过程泄露到输出中，无法干净分离
      Qwen2.5:7b 无思考模式，输出纯净，且 7b 参数量足够处理结构化事实
    - 系统 Prompt: [最高系统禁令] 严禁脑补，100% 基于已知事实
    - 输出三节中文报告:
      一、场景概览 (时长, 车辆数)
      二、车辆行为时序 (转弯车辆的运动状态)
      三、安全评估

  模块 3: 护栏防御 (guardrail_check)
    - 扫描 LLM 输出中的违禁词 (碰撞/事故/违章/伤亡 等12个关键词)
    - 否定感知: "未发现碰撞" 不触发 ("未" 在关键词前5字符内视为否定)
    - 触发后: 不做文本替换，而是重新调用 LLM 生成

  模块 4: 交互式问答 (interactive_qa_session)
    - 无状态 RAG: 每次问答独立基于场景上下文
    - 严格约束: "数据中未记录该信息" → 如实回答不知道

================================================================================
  七、关键设计决策与演进路径
================================================================================

  1. B2 (SAM2 路面分割) → 已移除
     原因: SAM2 的 road mask 对 Stage 2 的 2D 像素空间推理无直接贡献，
           且加载 856MB 模型拖慢管线，MVP 阶段砍掉

  2. A (OpenCLIP 信号灯颜色) → 已移除
     原因: 监控视角域偏移严重，零样本分类准确率不足，且信号灯语义对
           当前纯车辆轨迹推理非必需

  3. B3 (车道线检测) → 全部废弃
     原因: GANet/CLRerNet/CV Hough 在头顶监控视角下全部失效
           (训练集 BDD100K 为车载视角，域偏移导致 lane head 完全错位)
           YOLOPv2 检测头 100% 误分类为 truck

  4. Depth Anything v2: ViT-L → ViT-S
     原因: ViT-L (1.3GB) 加载慢 (~20s)，处理速度仅 1.3fps
           ViT-S (95MB) 秒加载，处理速度 ~4fps，精度损失在相对深度场景中可接受

  5. LLM: Qwen3.5 → Qwen2.5
     原因: Qwen3.5 的思考模式 (thinking mode) 会将英文推理链混入输出，
           无法通过简单正则分离；Qwen2.5:7b 无思考模式，输出纯净

  6. 轨迹平滑: conf加权30帧 → uniform加权15帧
     原因: 置信度与距离呈负相关（远处低、近处高），用 conf 做加权会
           导致平滑位置向近处（过去的）帧偏移，产生系统滞后

  7. 静止检测: 70%阈值 → 95%阈值
     原因: 70% 会把等红灯的车辆也标记为 Stationary，跳过 M2 分析
           95% 确保只有从头到尾静止的车辆才被覆盖

  8. 移除 longitudinal_trend 从 Stage 3 事实中
     原因: 深度衍生 "驶近/驶离" 基于相对深度变化，无空间参照物时噪声大，
          在只有 B1+C 的 v4 架构下不可靠
