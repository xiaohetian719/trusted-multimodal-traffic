================================================================================
  一、Stage 3 设计目标
================================================================================

Stage 3 是系统的最终输出层，将 Stage 2 的结构化 JSON 转化为自然语言报告。

核心约束:
  - LLM 不接触任何视觉数据（视频帧、深度图）
  - 报告 100% 基于 JSON 事实，每一句话必须可追溯
  - 护栏机制主动阻止 LLM "脑补"危险事件
  - "面向结果编程": 不把所有数据丢给 LLM，先做聚合统计

代码: narrative_agent.py (349行, v2 精简版)
模型: Ollama + Qwen2.5:7b (无思考模式)

================================================================================
  二、模块 1: 事实序列化
================================================================================

方法: serialize_graph_to_facts(json_data) → str

设计问题: Stage 2 的 ST-Graph JSON 包含数百辆车，每辆车约 80 字符，
如果全部列出 → ~11,000 字符 → LLM 上下文窗口压力大、容易跑偏。

解决方案: 聚合统计 + 只列转弯车辆

事实构建流程:
  1. 从 meta_info 提取总时长、总帧数
  2. 用 Counter 统计:
     - 转向分布: 直行/左转/右转 各多少辆
     - 运动状态: 加速/蠕行/巡驶/减速/静止 各多少辆
  3. 统计跟车关系数量
  4. 仅列出转弯车辆 (Left Turn / Right Turn)，每辆一行:
     car_XXXXX: 左转, 0.00s→5.60s, 加速

为什么只列转弯车辆:
  - 直行车辆占 81-85% → 列出来没有信息量
  - 转弯是交通分析中最有意义的语义
  - 从 134 辆全列 → 16 辆转弯，大幅压缩

为什么移除了 longitudinal_trend:
  - "驶近/驶离" 基于 Depth Anything v2 的相对深度变化
  - 在无空间参照物 (SAM2 路面已移除) 的情况下，噪声大
  - 继续使用会导致报告出现 "接近事件频发" 的误读
  - S2 JSON 中仍保留该字段，但 S3 不使用

最终事实文本: ~18 行，~1,000 字符，干净可控

═══════ 设计对比 ═══════

  旧方案 (505行版): 全量列出所有车辆 → 136行 → 11,374字符
  新方案 (349行版): 聚合统计 + 仅转弯车辆 → ~18行 → ~1,000字符
  效果: LLM 不再被海量数据淹没，输出质量明显提升

================================================================================
  三、模块 2: LLM 推理
================================================================================

方法: _call_ollama(prompt, system_prompt, stream, temperature)

系统 Prompt:
  "[最高系统禁令]:
   你是严谨的交通案件文书。所有叙事素材必须100%严格基于【绝对已知事实列表】，
   严禁脑补。若事实中未提及异常事件，报告中绝不允许编造。

   请严格按以下格式输出报告:
   ## 交通场景报告
   ### 一、场景概览
   - 监控时长: [时长]
   - 涉及车辆: [列表]
   ### 二、车辆行为时序
   按时间顺序列出每辆车的运动状态、转向语义。
   ### 三、安全评估
   基于事实给出安全评估。"

输出格式: 三节中文报告
  一、场景概览 — 总时长、车辆数、整体特征
  二、车辆行为时序 — 转弯车辆的运动状态（从 JSON 提取）
  三、安全评估 — 基于转向+运动状态的客观评估

推理参数:
  - model: qwen2.5:7b
  - temperature: 0.3 (低温度，减少随机性)
  - num_predict: 4096 (最大输出 token)
  - stream: True (流式输出到控制台)

流式输出设计:
  - 逐 token 打印到 stdout，让用户看到实时生成进度
  - 同时累积全部文本供护栏检查和保存
  - 支持 thinking token 收集（兼容 qwen3.5 的回退）

================================================================================
  四、模块 3: 护栏防御
================================================================================

方法: guardrail_check(llm_text, json_data) → (is_safe, text)

违禁词列表 (12个):
  "碰撞", "相撞", "撞上", "撞击", "事故", "车祸",
  "伤亡", "死", "重伤", "轻伤", "违法", "违章", "逃逸"

扫描逻辑:
  1. 对每个违禁词，在 LLM 输出文本中查找
  2. 检查违禁词前 5 个字符是否有否定前缀:
     "未", "没有", "无", "非", "不", "否", "未发现", "未检测到"
  3. 有否定前缀 → 不触发（"未发现碰撞" = 安全陈述）
  4. 无否定前缀 → 触发护栏

触发后的处理:
  - 不做文本替换（不篡改 LLM 输出）
  - 记录被触发的关键词
  - 重新调用 LLM 生成（最多重试 2 次）
  - 达到最大重试次数 → 使用最后一次输出作为兜底

设计理念:
  - 检测但不删除 → 保持文本完整性
  - 否定感知 → 避免误报
  - 重新生成而非替换 → 让 LLM 自己修正

与旧版 (505行) 的对比:
  - 旧版: 还有 _json_has_collision() 做 JSON 交叉核验
  - 新版: JSON 交叉核验已移除，因为:
    ① S2 JSON 中不存在碰撞/事故字段 → 核验无意义
    ② 违禁词扫描已足够有效
    ③ 简化代码，减少误报

================================================================================
  五、模块 4: 交互式问答
================================================================================

方法: interactive_qa_session(user_query, json_data) → str

特点:
  - 无状态: 每次问答独立，不携带历史对话（避免 LLM 跨轮次脑补）
  - RAG 风格: 先序列化场景事实，再基于事实回答
  - 严格约束: "数据中未记录该信息" → 如实回答不知道

与旧版 (505行) 的对比:
  - 旧版: chat_history 列表维护完整对话历史 → 336 tokens 可能跨轮次污染
  - 新版: 无状态，每次独立 → 更可控

================================================================================
  六、LLM 选型过程
================================================================================

  尝试 1: Qwen3.5:4b
    - 问题: 有思考模式 (thinking mode)，推理过程会泄露到输出中
    - 输出混合了中文报告和英文推理链，无法通过正则干净分离
    - _extract_response() 尝试提取纯回应，但效果不佳
    - 输出经常变成 ID 列表而非结构化报告

  尝试 2: Qwen3.5:9b
    - 参数量更大，但思考模式问题同样存在
    - 用户停止了该次测试

  最终选择: Qwen2.5:7b
    - 无思考模式 → 输出纯净，不混入推理链
    - 7b 参数量足够处理结构化事实文本
    - 生成速度快，约 53 秒生成完整报告 (S06_c046, 134辆车)
    - 报告格式稳定、可预测

================================================================================
  七、精简过程 (505行 → 349行, -31%)
================================================================================

  删除的内容:
    1. chat_history (对话历史) → 改为无状态 QA
    2. _json_has_collision() (JSON 交叉核验) → 违禁词扫描已足够
    3. _extract_response() 复杂正则 → 简化为 3 行直接提取
    4. per-vehicle 全量列出 → 聚合统计 + 仅转弯车辆
    5. longitudinal_trend 事实 → 深度衍生数据不可靠
    6. "notable vehicles" 宽泛条件 → 仅转弯+减速+跟车，上限 20

  保留的核心:
    1. serialize_graph_to_facts() — 事实序列化
    2. _call_ollama() / _stream_ollama() — LLM 调用
    3. guardrail_check() — 护栏扫描（简化但有效）
    4. interactive_qa_session() — 交互问答（改为无状态）
    5. generate_official_report() — 公开 API
