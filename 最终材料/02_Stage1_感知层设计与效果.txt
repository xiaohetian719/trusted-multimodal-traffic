================================================================================
  一、Stage 1 设计目标
================================================================================

Stage 1 的唯一职责：将原始视频帧转化为结构化的数值数据，不做任何语义判断。

设计约束:
  - 零语义解释: 不判断"这辆车在转弯"、"这里有人闯红灯"
  - 纯像素空间: 不做 3D 世界坐标投影（单目深度存在非线性畸变）
  - 轻量化: 砍掉所有不直接贡献车辆追踪的视觉任务
  - 快速: 目标处理速度 > 4fps (RTX 4060 Laptop 8GB)

当前架构 (v4):
  通道 B1: YOLO (BDD100K预训练) — 车辆检测 + 内置追踪
  通道 C:  Depth Anything v2 ViT-S — 单目深度估计

已移除的通道 (详见 05_探索与废弃方案记录.txt):
  B2: SAM2 路面分割 (~856MB 模型，对 2D 推理无直接贡献)
  A:  OpenCLIP 信号灯颜色 (域偏移严重，非当前必需)
  B3: 车道线检测 (车载→头顶监控域偏移，全部失效)

================================================================================
  二、通道 B1: YOLO 目标检测与追踪
================================================================================

模型: YOLO (BDD100K 预训练权重)
  - BDD100K 原生 10 类: pedestrian, rider, car, truck, bus, train,
    motorcycle, bicycle, traffic_light, traffic_sign
  - 仅保留 vehicle 类: cls_id ∈ {2, 3, 4, 5} (car/truck/bus/train)

追踪机制: YOLO 内置追踪 (persist=True)
  - 跨帧维持 track_id，追踪 ID 由 YOLO 内部分配
  - 相比独立 IoU Tracker (SimpleVehicleTracker)，减少了外部依赖

输出字段（每辆车）:
  - id:      整型 track_id（跨帧唯一）
  - cls_id:  BDD100K 类别 ID（2=car, 3=truck, 4=bus, 5=train）
  - class:   类别名称字符串 ("car", "truck", "bus", "train")
  - conf:    YOLO 检测置信度 (0-1)
  - cx, cy:  BBox 几何中心坐标 (像素)
  - bbox:    (x1, y1, x2, y2) 边界框

置信度阈值: 0.25
  - 低于此值的检测被直接丢弃
  - 置信度同时影响 Stage 2 Phase 1 的稳定池准入（avg_conf ≥ 0.4）

================================================================================
  三、通道 C: Depth Anything v2 单目深度估计
================================================================================

模型: Depth Anything v2 ViT-S
  - 参数量: ~95MB（对比 ViT-L 的 ~1.3GB）
  - 加载速度: 秒级（ViT-L 需 ~20s）
  - 处理速度: ~4fps（ViT-L 仅 ~1.3fps）

工作原理:
  1. 对每帧生成全图相对深度图 (H×W float32 矩阵)
  2. 输出值域: [0, 255] 归一化（0=远, 255=近）
  3. 在每辆车的 BBox 中心点 (cx, cy) 处采样深度值

相对深度的含义:
  - 数值大 = 像素亮 = 距离近（车在近处）
  - 数值小 = 像素暗 = 距离远（车在远处）
  - 是"相对"深度，不是绝对距离（米），不同画面的同一数值不可直接比较

ViT-S vs ViT-L 对比:
  - ViT-L (1.3GB): 深度图更精细，但加载慢、推理慢
  - ViT-S (95MB): 深度图稍粗糙，但在车辆级精度下足够使用
  - 对 Stage 2 的 M3 FSM 而言，深度趋势（变大/变小）比绝对数值更重要
  - 对比可视化: output/depth_compare/depth_vits_vs_vitl.avi

注意: 当前架构已废弃 longitudinal_trend（驶近/驶离判断），因为：
  - 相对深度在无空间参照物时噪声大
  - 摄像头角度、路面坡度等因素导致深度变化不纯粹反映车辆运动
  - Stage 2 仍计算该字段，但 Stage 3 不再喂给 LLM

================================================================================
  四、质心深度对齐方案
================================================================================

设计方案选择: 2D 像素空间 + 相对深度采样（非 3D 投影）

为什么不用 3D 投影:
  - 单目深度估计存在严重的非线性畸变
  - 将像素坐标反投影到世界坐标需要相机内参和外参
  - 头顶监控视角的相机标定困难（无标定板/已知尺寸参照物）
  - 2D 像素空间足以支撑后续的轨迹分析和转向判断

质心计算:
  cx = (x1 + x2) / 2    (BBox 水平中心)
  cy = (y1 + y2) / 2    (BBox 垂直中心)

深度采样:
  depth = depth_map[cy][cx]    (直接从深度图取像素值)
  采样前进行边界裁剪: 0 ≤ cx < W, 0 ≤ cy < H

为什么不追踪 BBox 角点:
  - "车轮接地投影"方案需要准确的 3D 车轮位置
  - YOLO 的 BBox 不包含车轮位置信息
  - 质心已足够表征车辆整体运动趋势

================================================================================
  五、输出格式与数据量
================================================================================

Frame Dict 格式 (v4):
  {
    "frame_id": int,         // 帧序号 (从 0 开始)
    "timestamp": float,      // 时间戳 (秒)
    "objects": [             // 当前帧的所有车辆
      {
        "id": int,           // track_id
        "cls_id": int,       // BDD100K 类别 ID
        "class": str,        // 类别名称
        "conf": float,       // 检测置信度
        "cx": int,           // 中心 x (像素)
        "cy": int,           // 中心 y (像素)
        "depth": float       // 相对深度值
      }
    ]
  }

数据量 (10fps 采样):
  视频          | 帧数  | Pickle 大小 | 车辆数(unique)
  --------------|-------|-------------|---------------
  S05_c028      | ~2000 | ~7.6 GB     | ~636
  S06_c041      | ~1270 | ~2.3 GB     | ~422
  S06_c043      | ~1250 | ~2.3 GB     | ~430
  S06_c046      | ~2000 | ~1.8 GB     | ~233

注意: pickle 文件很大 (1-8GB)，因为每帧都保存了完整的车辆列表。

================================================================================
  七、关键设计决策与参数选取依据
================================================================================

  决策 1: 为什么只保留 B1 + C 两个通道？
    依据: "奥卡姆剃刀" — 砍掉不直接贡献车辆追踪的视觉任务
    B2 (SAM2 路面分割) 的 road mask 在 2D 像素空间中对轨迹分析无增益
    A (OpenCLIP 信号灯) 的域偏移导致准确率不足
    B3 (车道线) 在头顶监控视角下完全失效
    结果: 管线从 4 通道精简到 2 通道，内存占用降低 ~5GB

  决策 2: 为什么使用 ViT-S 而非 ViT-L？
    依据: 速度/精度权衡
    ViT-L ~1.3fps → 不适合实时处理
    ViT-S ~4fps → 可接受
    深度图的"趋势"比"绝对值"更重要（FSM 只关心变大还是变小）

  决策 3: 为什么只保留 vehicle 类别？
    依据: Stage 2 只推理车辆轨迹
    行人/信号灯/标志牌在当前 MVP 阶段不进入时空因果推理
    预过滤减少了 S2 的数据处理量

  决策 4: conf_thresh = 0.25 的依据？
    依据: 平衡召回率和精度
    更低 → 更多误检进入追踪
    更高 → 远处车辆（置信度自然较低）被丢弃
    0.25 是 YOLO 默认值，在 BDD100K 场景下表现稳定
    同时 Stage 2 Phase 1 有第二层过滤 (avg_conf ≥ 0.4)

  决策 5: 为什么不做 3D 投影？
    依据: 单目深度存在严重的非线性畸变
    不做 → 避免引入不可控的误差
    2D 像素空间 + 相对深度 → 足以支撑微分几何转向判断
    Stage 2 的 M2 曲率积分只需 (cx, cy) 序列
